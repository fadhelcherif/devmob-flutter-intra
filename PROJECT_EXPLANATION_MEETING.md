## 1) What this app is

`devmobi_flutter_intra` is an internal social app for a company/university team.
Main features:

- user authentication (login/signup)
- profile management
- posts (global and group posts)
- comments and likes
- groups (public/private)
- one-to-one messaging
- group messaging
- image/document sharing in chats and posts

---

## 2) Project structure (easy map)

At root level:

- `lib/` -> all Dart app code
- `android/`, `ios/`, `web/`, `windows/`, `macos/`, `linux/` -> platform-specific files
- `pubspec.yaml` -> dependencies and app config
- `firebase.json` -> Firebase project + rules/index config
- `firestore.rules` -> Firestore security rules
- `storage.rules` -> Firebase Storage security rules

Inside `lib/`:

- `main.dart` -> app entry point
- `firebase_options.dart` -> generated Firebase config for each platform
- `models/` -> data models (User, Post, Message, Group, Comment)
- `services/` -> business logic + Firebase/Cloudinary access
- `views/` -> UI screens (auth, home, chat, group, profile)
- `providers/` -> state management (Theme, GroupMessage)
- `widgets/` -> reusable UI components

---

## 3) How app startup works

Flow when app opens:

1. `main.dart` calls `Firebase.initializeApp(...)` using `DefaultFirebaseOptions.currentPlatform`.
2. App starts with `SplashScreen`.
3. `SplashScreen` checks `FirebaseAuth.instance.currentUser`.
4. If user exists -> go to `FeedScreen`.
5. If not logged in -> go to welcome/login flow.

So Firebase is initialized once at startup, then every service uses Firebase instances.

---

## 4) How Firebase connection was made

This project uses FlutterFire setup.

Key files:

- `lib/firebase_options.dart` (generated by FlutterFire CLI)
- `android/app/google-services.json` (Android config)
- `firebase.json` (links Firestore rules/indexes)

`firebase_options.dart` contains:

- `projectId: devmob-flutter-intra`
- app IDs for android/ios/web/windows/macos
- api keys and other options

At runtime, app connects like this:

```dart
await Firebase.initializeApp(
  options: DefaultFirebaseOptions.currentPlatform,
);
```

After initialization:

- `FirebaseAuth.instance` handles login/signup
- `FirebaseFirestore.instance` handles database reads/writes

## 5) Core architecture pattern

The app follows a simple layered structure:

- UI layer: `views/`
- logic/data layer: `services/`
- schema layer: `models/`

Example:

- `ChatDetailScreen` (UI) calls `ChatService.sendMessage(...)`
- `ChatService` writes Firestore documents
- `MessageModel` defines the message shape

This keeps UI code cleaner and puts Firebase logic in one place.

## 6) Database structure (Firestore)

Main collections used in this project:

## `users/{uid}`

Stores profile + role.
Typical fields:

- `uid`, `email`, `name`
- `bio`, `profileImageUrl`
- `role` (`admin` or `employee`)
- `createdAt`

## `posts/{postId}`

Stores posts (normal or group-related).
Typical fields:

- `userId`, `userName`, `userImage`
- `content`
- optional: `groupId`, `groupName`
- optional: `imageUrl`, `documentUrl`, `documentName`
- `likes` (array of user IDs)
- `commentsCount`
- `createdAt`

Subcollection:

- `posts/{postId}/comments/{commentId}`

Comment fields:

- `id`, `postId`, `userId`, `userName`, `userImage`, `content`, `createdAt`

## `groups/{groupId}`

Stores group metadata.
Typical fields:

- `name`, `description`
- `createdBy`, `creatorName`, `creatorImage`
- `isPublic`
- `members` (array of user IDs)
- `pendingMembers` (for private join requests)
- `createdAt`
- chat preview fields: `lastMessage`, `lastMessageTime`, `lastMessageSender`

Subcollection for group messages:

- `groups/{groupId}/messages/{messageId}`

## `chats/{chatId}` (one-to-one chats)

Stores direct conversation metadata.
Typical fields:

- `participants` (2 user IDs)
- `lastMessage`
- `lastMessageTime`

Subcollection:

- `chats/{chatId}/messages/{messageId}`

Message fields include:

- `chatId` or `groupId`
- `senderId`, `senderName`, `senderImage`
- `content`, `timestamp`
- optional: `isImage`, `imageUrl`
- optional: `isDocument`, `documentUrl`, `documentName`

## 7) How direct chat works (step by step)

Files mainly involved:

- UI: `views/chat/chat_list.dart`, `views/chat/chat_detail.dart`
- Logic: `services/chat_service.dart`
- Data model: `models/message_model.dart`

Flow:

1. User opens Members/Discover and clicks message icon.
2. App opens `ChatDetailScreen` with receiver user ID.
3. `ChatService.getChatId(userA, userB)` creates a deterministic ID (sorted IDs joined with `_`).
4. `ensureDirectChatExists(...)` creates/merges `chats/{chatId}` if needed.
5. Sending text/image/document adds a doc to `chats/{chatId}/messages`.
6. Chat preview (`lastMessage`, `lastMessageTime`) is updated in `chats/{chatId}`.
7. Message list listens in real time via Firestore stream ordered by timestamp.

Important point to present:

- same two users always produce the same `chatId`, so they always return to the same conversation.

## 8) How group chat works

Files mainly involved:

- UI: `views/group/group_chat_screen.dart`
- Logic: `services/chat_service.dart`, `services/group_service.dart`
- Provider: `providers/group_message_provider.dart`

Flow:

1. User opens a group chat screen.
2. Screen subscribes to `groups/{groupId}/messages` stream.
3. When user sends message, app writes new message doc under group messages.
4. Group doc is also updated with `lastMessage` and `lastMessageTime`.
5. Attachments (image/document) are uploaded first, then URL saved in message document.

---

## 8.1) Super simple: where each message goes

This is the easiest way to explain it:

- direct message (between 2 users) -> saved in `chats/{chatId}/messages/{messageId}`
- group message (inside a group) -> saved in `groups/{groupId}/messages/{messageId}`

What happens when user sends a message:

1. UI screen calls `ChatService` send function.
2. If message has file/image, file is uploaded to Cloudinary first.
3. Firestore message document is created with sender info + timestamp + content/url.
4. Parent document preview is updated:
  - direct chat updates `chats/{chatId}.lastMessage`
  - group chat updates `groups/{groupId}.lastMessage`
5. Other users see the new message instantly through Firestore stream listeners.

How messages are read in real time:

- direct chat screen listens to `chats/{chatId}/messages` ordered by `timestamp`
- group chat screen listens to `groups/{groupId}/messages` ordered by `timestamp`

Extra detail (if teacher asks):

- direct chat uses one stable `chatId` made from both user IDs sorted alphabetically, so conversation is always the same thread
- group chat uses `groupId` directly, so all group members read/write in the same message subcollection

## 9) Media and files (not only Firebase)

Media upload is handled by `StorageService` with Cloudinary:

- images uploaded to Cloudinary folders
- documents uploaded as raw files to Cloudinary
- returned URL is stored in Firestore message/post docs

So data architecture is:

- Firebase Auth -> authentication
- Firestore -> app data and chat messages
- Cloudinary -> actual file hosting for images/documents

`storage.rules` exists for Firebase Storage but current service code uses Cloudinary for uploads.

## 10) Security rules summary

From `firestore.rules`:

- only authenticated users can access most data
- users can only update/delete their own profile docs
- post/comment creation checks sender identity
- direct chat access is only for participants
- direct message creation enforces sender and chat consistency
- group message read/write requires group membership

This means unauthorized users cannot read private chats or post as someone else.

## 11) Typical end-to-end scenarios

## A) Login and open feed

1. Login screen calls `AuthService.login()`.
2. Firebase Auth verifies credentials.
3. App loads user profile from `users/{uid}`.
4. Navigate to `FeedScreen`.

## B) Create a post

1. UI collects text/media info.
2. `PostService.createPost()` creates doc in `posts`.
3. Feed stream updates automatically.

## C) Add comment

1. `CommentService.createComment(...)` creates comment subdoc.
2. Post `commentsCount` increments.
3. Post detail screen updates in real time.

## D) Send direct message

1. Upload media if needed (Cloudinary).
2. `ChatService.sendMessage/sendImageMessage/sendDocumentMessage` writes in `chats/{chatId}/messages`.
3. Chat list preview updates from `chats` collection.
